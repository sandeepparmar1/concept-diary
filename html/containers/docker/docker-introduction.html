Docker — Core Concepts

Images, containers, Dockerfile, volumes and how Docker fits into the dev workflow
Notebook • Docker

Docker packages an application and all of its dependencies into a lightweight, portable container. Containers share the host kernel, so they boot fast and use fewer resources than full VMs. Below are the core ideas you should remember and link out to as individual topic pages.

Images

Immutable blueprints made of layers. Build with a Dockerfile and push to a registry.

Containers

Runtime instances of images. Start, stop, and inspect the process & filesystem.

Dockerfile

Step-by-step instructions (FROM, RUN, COPY, CMD) used to build images.

Registry

Store and distribute images (Docker Hub, GHCR). Tag & push from CI.

Volumes

Persistent storage outside containers for databases and stateful services.

Networks

Connect containers with user-defined networks; map ports and isolate traffic.

Quick Commands

Build — docker build -t myapp:1.0 .

Run — docker run -d -p 8080:80 myapp:1.0

List — docker ps -a

Tip: Use small base images and multi-stage builds for lighter images.

Docker Architecture

Client → Daemon → Kernel → Hardware. The daemon (dockerd) talks to the OS to create namespaces, cgroups, and attach filesystems to containers.

Core Pieces

  • Docker Engine: Builds and runs containers.
  • Docker Daemon: Background service that manages state.
  • Docker CLI: User interface to talk with daemon.

DevOps Workflow

  1. Build image from Dockerfile.
  2. Test containers locally or in CI.
  3. Push to registry (CI → registry).
  4. Deploy pull & run in production; orchestrate with Kubernetes.

Notes

Containers are ephemeral. Use volumes for persistent data. Security: run minimal privileges and scan images for vulnerabilities.

Simple Dockerfile Example

FROM openjdk:17-jdk-slim
WORKDIR /app
COPY target/myapp.jar /app/
CMD ["java", "-jar", "myapp.jar"]
